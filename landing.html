01 — Problem

THE PROBLEM
WITH BRIDGES

⛓
Chain Isolation
Avalanche subnets are isolated execution environments. Assets native to C-Chain cannot natively exist on your custom L1 — users are locked into a single chain.
🔓
Trust-Heavy Bridges
Most bridges rely on centralized multisigs or off-chain validators. A single point of compromise means total loss — as seen in $2B+ in bridge hacks since 2021.
🐌
Slow Finality
Traditional cross-chain transfers wait for 20+ block confirmations and manual relaying. Users sit through 10–30 minute waits for simple token movements.
💸
Excessive Fees
Third-party bridge protocols charge 0.05–0.3% per transfer, plus gas on both chains. For frequent or small transfers, fees become prohibitive for users.
02 — Solution

AVABRIDGE
CHANGES THIS

AvaBridge uses Avalanche Teleporter — a native cross-chain messaging protocol built into the Avalanche network layer. Instead of trusting a third-party bridge operator, messages are validated by Avalanche's own validator set via the Warp messaging standard.

On the source chain, tokens are locked in a non-custodial smart contract. Teleporter relayers detect the lock event and deliver a verified message to the destination chain, where wrapped tokens are minted 1:1 to the recipient.

The return path burns wrapped tokens on the subnet and unlocks originals on C-Chain — a complete lock-mint / burn-release cycle with no intermediaries, no custodians, no trust required.

03 — Features

WHAT IT
DOES

01
Native Teleporter
Uses Avalanche's built-in ICM (Interchain Messaging) instead of third-party relayers. Warp messages are validated by the same validator set securing the network.
Security
02
Lock / Mint Model
Source tokens are locked, never burned. Wrapped tokens are minted 1:1 on the destination chain. Total supply is always fully backed — no fractional reserve risk.
Design
03
Two-Way Bridge
Full round-trip support. Bridge forward from C-Chain to subnet, or bridge back by burning wrapped tokens and unlocking originals. Complete freedom of movement.
UX
04
Event-Driven
Every action emits on-chain events: BridgeInitiated, TokensMintedOnDestination, ReturnInitiated, BridgeCompleted. Full auditability and indexing support.
Transparency
05
Zero Protocol Fee
The bridge contracts charge no fees. Users only pay gas on each chain. Open relayer model means anyone can relay messages — no fee extraction at the protocol level.
Economics
06
Multi-Destination
The sender contract supports registering multiple destination chains. One deployment on C-Chain can serve N subnets — extendable without redeployment.
Scalability
07
Permissioned Minting
BridgedToken uses an access-controlled mint/burn interface — only the BridgeReceiver can mint. Protects against inflation attacks even if the frontend is compromised.
Security
08
Full-Stack UI
A production-grade frontend with MetaMask integration, live balance display, auto-approval flow, transaction log, event feed, and block explorer links.
DX
09
Emergency Recovery
Owner-only token recovery function on the sender contract. If a bridge message fails irrecoverably, the owner can return locked tokens — no user funds are permanently lost.
Safety

05 — How It Works

STEP BY
STEP

1
Source → Destination
User Approves Token Spend
The user grants the TokenBridgeSender contract permission to pull their ERC20 tokens. The bridge UI handles this automatically — no manual approve needed.
await token.approve(TokenBridgeSender.address, amount)
2
Source Chain
Tokens Are Locked
The sender contract pulls the tokens from the user, locks them, increments totalLocked, and emits BridgeInitiated. Funds are safely held — not burned, not moved.
sender.bridgeTokens(destinationChainID, recipient, amount)
// emits: BridgeInitiated(chainID, from, to, amount, msgID)
3
Teleporter Layer
Warp Message Delivered
Avalanche Teleporter relayers detect the sendCrossChainMessage call and deliver a Warp-signed message to the destination chain's BridgeReceiver within ~15 seconds. No external relayer needed.
teleporter.sendCrossChainMessage(TeleporterMessageInput{
destinationBlockchainID, destinationAddress, payload
})
4
Destination Chain
Wrapped Tokens Are Minted
The BridgeReceiver validates the message origin, decodes the payload, and calls BridgedToken.mint() — creating exactly the same amount of wrapped tokens for the recipient.
receiver.receiveTeleporterMessage(srcChainID, origin, msg)
// emits: TokensMintedOnDestination(recipient, amount)
5
Return Journey
Burn → Unlock
When bridging back, the user approves + calls returnTokens(). The receiver burns the wrapped tokens and sends a Teleporter message back to the sender, which releases the original locked tokens.
receiver.returnTokens(amount, sourceRecipient)
// burns bToken → sends Teleporter → unlocks ERC20 on source
06 — Contracts

4 SMART
CONTRACTS

TokenBridgeSender
C-Chain
The entry point on the source chain. Handles ERC20 locking, calls Teleporter's sendCrossChainMessage, and listens for return messages to release locked tokens. Supports multiple destination chains via registerDestination().
Events
BridgeInitiated(destChainID, sender, recipient, amount, msgID)
BridgeCompleted(srcChainID, recipient, amount)
DestinationRegistered(chainID, receiverContract)
TokensRecovered(to, amount)
TokenBridgeReceiver
Subnet
Lives on the destination subnet. Implements ITeleporterReceiver to receive Warp messages, validates origin, and calls BridgedToken.mint(). Also handles the return path by burning tokens and firing a Teleporter message back.
Events
TokensMintedOnDestination(recipient, amount, srcChainID)
ReturnInitiated(burner, srcRecipient, amount, msgID)
SourceBridgeSet(chainID, bridge)
BridgedToken
Subnet
A minimal ERC20 with permissioned mint/burn. Only the BridgeReceiver address can call mint() or burn(). Standard transfer, approve, and transferFrom are fully public. Decimals match the source token (18).
Events
TokensMinted(to, amount)
TokensBurned(from, amount)
Transfer(from, to, amount) — ERC20 standard
BridgeUpdated(oldBridge, newBridge)
ITeleporterMessenger
Both Chains
Simplified interface to Ava Labs' Teleporter protocol. The actual contract is deployed and maintained by Ava Labs at the same address on every Avalanche chain. Provides sendCrossChainMessage() and messageReceived().
Address (all chains)
0x253b2784c75e510dD0fF1da844684a1aC0aa5fcf
Key Methods
sendCrossChainMessage(TeleporterMessageInput)
messageReceived(srcChainID, msgID) → bool
